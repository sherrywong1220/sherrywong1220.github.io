《图解HTTP》学习笔记（六）——HTTP首部

2017-03-02

章节回顾

  • 首部字段给浏览器和服务器提供报文主体大小、所使用的语言、认证信息等内容。
  • 其实在学习笔记（三）中已经简单讲解了什么是HTTP报文，什么是报文主体和报文首部，以及首部的构成。
  • 大家可以先回顾一下《图解HTTP》学习笔记（三）——HTTP报文内的HTTP信息的内容。

首部字段

4种首部字段类型

  • 通用首部字段
    请求报文和响应报文两方都会使用的首部。
  • 请求首部字段
    从客户端向服务器端发送请求报文时使用的首部。补充了请求的附加内容、客户端信息、响应内容相关优先级等信息。
  • 响应首部字段
    从服务器端向客户器端发送请求报文时使用的首部。补充了响应的附加内容也会要求客户端附加额外的内容信息。
  • 实体首部字段
    针对请求报文和响应报文的实体部分使用的首部。补充了资源内容更新时间等与实体有关的信息。

《图解HTTP》学习笔记（五）——与HTTP协作的Web服务器

2017-03-01

用单台虚拟主机实现多个域名

  • HTTP/1.1规范允许一台HTTP服务器搭建多个Web站点，这是因为利用了虚拟主机（Virtual Host,又称虚拟服务器）的功能。
  • 即使物理层面只有一台服务器，但只要使用虚拟主机的功能，则可以假设已具有多台服务器。
  • 如果一台服务器托管了不同的域名，收到对它们的访问请求后，DNS服务解析域名会得到相同的IP地址。
  • 在相同的IP地址下，由于虚拟主机可以寄存多个不同主机名和域名的Web网站，因此在发送HTTP请求时，必须在Host首部内完整指定主机名或域名的URI。

通信数据转发程序：代理、网关、隧道

  • 通信数据转发程序和服务器可以将请求转发给通信线路上的下一站服务器，并且能接受从那台服务器发送的响应，再发送给客户端。

代理

  • 代理扮演了位于服务器和客户端“中间人”的角色，接收由客户端发送的请求并转发给服务器，同时也接收服务器返回的响应并转发给客户端。
  • 代理不改变请求的URI，会直接发送给前方持有资源的目标服务器（源服务器）。
  • 在HTTP通信过程中，可级联多台代理服务器。转发时，需要附加Via首部字段以标记经过的主机信息。
  • 优点：利用缓存技术减少网络带宽的流量，组织内部针对特定网站的访问控制，以获取访问日志为主要目的，等等。
  • 使用代理的两类方法：一种是是否使用缓存，另一种是是否会修改报文。
  • 缓存代理
    • 代理转发响应时，缓存代理（Caching Proxy）会预先将资源的副本（缓存）保存在代理服务器上。
    • 当代理再次接收到对相同资源的请求时，就可以不从源服务器那里获取资源，而是将之前缓存的资源作为响应返回。
  • 透明代理
    • 转发请求或响应时，不对报文做任何加工的代理类型被成为透明代理（Transparent Proxy），反之，成为非透明代理。

《图解HTTP》学习笔记（四）——返回结果的HTTP状态码

2017-03-01

状态码告知从服务器端返回的请求结果

  • 状态码的职责是当客户端向服务器发送请求时，描述返回的请求结果。
  • 借助状态码，用户可以知道服务器端是正常处理了请求，还是出现了错误。
  • 状态码类别
    | 状态码 | 类别 | 原因短语 |
    | —- | ———————- | ————- |
    | 1XX | Informational（信息状态码） | 接受的请求正在处理 |
    | 2XX | Success（成功状态码） | 请求正常处理完毕 |
    | 3XX | Redirection（重定向状态码） | 需要进行附加操作以完成请求 |
    | 4XX | Client Error（客户端错误状态码） | 服务器无法处理请求 |
    | 5XX | Server Error（服务器错误状态码） | 服务器处理请求出错 |

14个具有代表性的状态码

2XX成功

  • 200 OK
    • 表示从客户端发来的请求在服务器端被正常处理了。
  • 204 No Content
    • 表示服务器已成功处理请求，但返回的响应报文中不含实体的主体部分。也不允许返回任何实体的主体。
    • 一般在需要从客户端往服务器发送信息，而对客户端不需要发送新信息内容的情况下使用。
  • 206 Partial Content
    • 表示客户端进行了范围请求，而服务器成功执行了这部分的GET请求。
    • 响应报文中包含由Content-Range指定范围的实体内容。

《图解HTTP》学习笔记（三）——HTTP报文内的HTTP信息

2017-01-13

HTTP报文及其结构

  • 用于HTTP协议交互的信息被称为HTTP报文。
  • 请求端：请求报文。
  • 响应端：响应报文。
  • HTTP报文：分为报文首部和报文主体，中间由空行分割。
  • 一定要有报文首部，不一定要有报文主体。
  • 请求报文首部：请求行，请求首部字段，通用首部字段，实体首部字段，其他。
  • 响应报文首部：状态行，响应首部字段，通用首部字段，实体首部字段，其他。
  • 请求行：用于请求的方法，请求URI，HTTP版本。
  • 状态行：HTTP版本，响应结果的状态码，原因短语
  • 首部字段：包含表示请求和响应的各种条件和属性的各类首部。
  • 其他：可能包含HTTP的RFC里未定义的首部（Cookie等）。

《图解HTTP》学习笔记（一）——了解Web及网络基础

2016-12-14

使用HTTP协议访问Web

当我们在浏览器地址栏输入URL时，Web页面是如何呈现的呢？

根据Web浏览器地址栏中指定的URL，浏览器从Web服务器端获取文件资源（resource）等信息，从而显示出Web页面。

像这种通过发送请求获取服务器资源的Web浏览器等，都可以成为客户端（client）。

Web使用一种名为HTTP(HyperText Transfer Protocol，超文本传输协议)的协议作为规范，完成从客户端到服务器端等一系列运作流程。协议指规则的约定。Web是建立在HTTP协议上通信的。

作为Web文档传输协议的HTTP，它的版本更新非常非常慢，现在刚到HTTP/2.0。

当年HTTP协议的出现主要是为了解决文本传输的难题。由于协议本身非常简单，于是在此基础上设想了很多应用方法并投入了实际使用。现在HTTP协议已经超出了Web这个框架的局限，被运用到了各种场景里。
